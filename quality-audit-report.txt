HIGH SEVERITY
============================================================

H-1: Auto-resume resurrects manually-paused campaigns
File: src/app/api/clips/[id]/review/route.ts, line 368
Also: src/app/api/admin/fix-budget/route.ts, lines 188-215
Description: The auto-resume guard checks lastBudgetPauseAt != null,
but this field persists even after an owner manually pauses a campaign.
An owner who manually pauses a campaign that still has lastBudgetPauseAt
from a prior budget pause will see it auto-resumed when any clip is
rejected or when fix-budget runs.
Fix: When owner manually pauses (via PATCH status: PAUSED), clear
lastBudgetPauseAt. Or add a dedicated isBudgetPaused boolean field.

============================================================
MEDIUM SEVERITY
============================================================

M-1: oldEarnings stale snapshot in tracking.ts
File: src/lib/tracking.ts, lines 264, 377
Description: oldEarnings captures clip.earnings from the job-load
snapshot (potentially minutes stale). earningsChanged compares against
this stale value, not the live DB value after the transaction.
Can cause false positive/negative on user stats sync.
Fix: After transaction, re-read clip earnings or pass the DB value
out of the transaction closure.

M-2: streakRestoredAt hides failed days permanently
File: src/lib/gamification.ts, lines 174-179
Description: After streak restore, 36h grace window suppresses day
evaluation. lastActiveDate is set to yesterday. Days that fail during
the grace window are never re-evaluated after it expires, because the
cursor starts from lastActiveDate+1 (which is today). The restored
streak value persists permanently even if user posted nothing.
Fix: Remove the streakRestoredAt skip — lastActiveDate=yesterday
already provides the needed grace. Or document as intentional.

M-3: Stale closure in earnings SSE debounce handler
File: src/app/(app)/earnings/page.tsx, lines 71-83
Description: SSE handler closes over selectedCampaigns state. If user
changes campaign filter within 500ms debounce, the queued fetchData
call uses the old campaign selection. Data eventually corrects on the
next event or re-registration.
Fix: Use a ref for selectedCampaigns in the handler closure.

M-4: TODAY cell w-8 overflow on 320px screens
File: src/app/(app)/progress/page.tsx, line 318
Description: Fixed w-8 (32px) on TODAY cell may overflow the grid
column on very small screens where 10 columns = ~28px each.
Fix: Use w-full instead of w-8, let grid control width.

LOW SEVERITY
============================================================

L-1: fix-budget third pass resumes any paused campaign
File: src/app/api/admin/fix-budget/route.ts, lines 188-215
Description: Same underlying issue as H-1. The third pass resumes
ALL paused campaigns with lastBudgetPauseAt and available budget,
including manually-paused ones.
Fix: See H-1.

L-2: Payout bonusPercentage only stores level bonus
File: src/app/api/payouts/route.ts, lines 84-86
Description: bonusPercentage field on User is the level bonus only.
Payout record stores this as bonusPercent, missing streak+PWA bonus.
Fee calculation is correct (fee is on requested amount, not earnings).
Only affects display in admin payout review.
Fix: Fetch streak and PWA bonus and sum locally, or rename field.

L-3: Both SSE endpoints register with broadcast
Files: src/app/api/chat/sse/route.ts, src/app/api/notifications/sse/route.ts
Description: Both register controllers for the same userId.
broadcastToUser sends events to both connections. Frontend handlers
may fire twice per event in the same tab.
Fix: Namespace events or use single SSE endpoint.

L-4: lastActiveDate not advanced past streak break
File: src/lib/gamification.ts, lines 252, 268
Description: After a streak break, lastActiveDate still points to
the last passed date (before the break). Next evaluation re-walks
the failed day(s). Harmless but wastes DB queries.
Fix: Set lastPassedDate = null on break, or advance to cursor.

L-5: Earnings "Available" is all-time while other cards are timeframe-filtered
File: src/app/(app)/earnings/page.tsx, lines 104-113
Description: "Total earned" and "Approved" show timeframe-filtered
values, but "Available for payout" shows all-time balance. Semantic
inconsistency that may confuse users.
Fix: Label as "Available for payout (all time)" or separate visually.

L-6: Chart colors hardcoded as #2596be, not synced to CSS variable
Files: src/components/earnings/EarningsChart.tsx,
src/components/ui/simple-chart.tsx, src/app/(app)/admin/analytics/page.tsx
Description: If accent color changes in CSS, charts will be out of
sync. Currently matches but is fragile.
Fix: Read --color-accent at runtime or export from theme constant.

L-7: Help page sections array recreated every render
File: src/app/(app)/help/page.tsx, lines 35-181
Description: JSX content nodes in sections array create new references
on each render. Practically harmless for a static page.
Fix: Move outside component or wrap in useMemo.

L-8: Admin clips page: double refresh on review action
File: src/app/(app)/admin/clips/page.tsx, lines 94-98
Description: Optimistic local update + immediate load() = redundant.
Causes two renders and a flash of optimistic state.
Fix: Remove optimistic update or remove load() — pick one.

L-9: TimeframeSelect buttons don't wrap on 320px screens
File: src/components/ui/timeframe-select.tsx, lines 17-34
Description: 5 buttons in a non-wrapping flex. Overflows on very
small screens when rendered alongside other filter controls.
Fix: Add flex-wrap to the button container.
