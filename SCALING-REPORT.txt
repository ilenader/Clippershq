2.5 Overlapping Cron Runs
    - No guard against overlapping cron executions
    - If cron run A doesn't finish in 5 min, cron run B starts
    - Both query nextCheckAt <= now(), so they grab DIFFERENT clips (already updated)
    - Risk: double-processing of clips currently being processed by run A
    - Mitigation: Serializable transactions prevent duplicate earnings writes
    - Worst case: wasted Apify calls, no data corruption

2.6 When Tracking Falls Behind
    - At ~5000+ active clips: cron processes 100 per run, 288 runs/day = 28,800 checks/day
    - If clips need 7000+ checks/day: some clips won't be checked on schedule
    - Clips just get checked on the next available run — delays, not data loss
    - Solution: increase take limit, add dedicated tracking worker


3. REAL-TIME (SSE)
================================================================================

3.1 How SSE Works
    - Client connects to /api/notifications/sse [src/app/api/notifications/sse/route.ts]
    - Server creates ReadableStream, stores controller in global Map [src/lib/sse-broadcast.ts:18]
    - Map structure: userId -> Set<ReadableStreamDefaultController>
    - Every 2 seconds: queries notification count from DB [sse/route.ts:59]
    - Every 15 seconds: sends heartbeat comment [sse/route.ts:68]
    - On events: broadcastToUser enqueues data to all user's controllers [sse-broadcast.ts:40]

3.2 Why SSE Is Unreliable on Vercel Serverless
    - Each Vercel function instance has its OWN in-memory Map
    - If user A connects to instance 1, and tracking runs on instance 2,
      instance 2's broadcastToUser has NO reference to user A's controller
    - SSE only works if the broadcast happens on the SAME instance as the connection
    - On Vercel Pro: functions can have 1-minute idle timeout, killing SSE streams
    - Result: SSE connections frequently drop and reconnect

3.3 Concurrent SSE Connections
    - No explicit connection limit per user or global [src/lib/sse-broadcast.ts:18]
    - Each connection = 1 open HTTP response + DB query every 2 seconds
    - 100 users = 100 SSE connections = 50 DB queries/second (notification counts)
    - 500 users = 500 connections = 250 queries/second = DB OVERLOAD
    - 1000 users = BROKEN without optimization

3.4 Events Broadcast
    Event               | Source                  | Recipients
    ------------------- | ----------------------- | ----------
    connected           | SSE route               | Connecting user
    notif               | SSE polling loop        | Each user (their own count)
    clip_updated        | tracking.ts:479         | Clip owner (userId)
    earnings_updated    | tracking.ts:480         | Clip owner (userId)
    tracking_progress   | tracking.ts:615-669     | All OWNER users

3.5 Fallback When SSE Fails
    - Frontend polling via useAutoRefresh hooks
    - Dashboard: 60-second polling [src/app/(app)/dashboard/page.tsx]
    - Clips: 60-second polling [src/app/(app)/clips/page.tsx]
    - All pages listen for SSE custom events on window, fall back to timer

3.6 SSE Scaling
    Users | SSE Works? | DB Load (queries/sec) | Notes
    ----- | ---------- | --------------------- | -----
    100   | Partially  | ~50/sec               | Some connections on wrong instance
    500   | Unreliable | ~250/sec              | DB notification count hammered
    1000  | BROKEN     | ~500/sec              | Need Redis pub/sub or Pusher

9. WORST CASE SCENARIOS
================================================================================

9.1 Scenario: 10,000 Signups in One Day
    WHAT BREAKS FIRST: Database connections
    - 10,000 users hitting dashboard = ~10,000 gamification queries
    - Each gamification call = 11-15 DB queries = ~150,000 queries
    - Supabase connection pool (200 max) overwhelmed within minutes
    - SSE connections: 10,000 * notification polling = 5,000 queries/second
    FIX: Add Redis caching for gamification, disable SSE polling, add CDN

9.2 Scenario: 1000 Clips Submitted in One Hour
    WHAT BREAKS FIRST: Cron tracking capacity
    - 1000 new clips = 1000 tracking jobs created
    - Cron processes 100 clips per 5-min run = ~1200/hour max capacity
    - Will keep up, but barely. Queue will build up during peak.
    - Apify: 1000 calls in first hour, then decreasing. Fine on Personal plan.
    FIX: Increase cron take limit from 100 to 200+

9.3 Scenario: 500 Users Hit Site at Same Time
    WHAT BREAKS FIRST: SSE notification polling
    - 500 SSE connections = 250 DB queries/second just for notifications
    - Plus page load queries: 500 * 20 = 10,000 queries in first 5 seconds
    - Supabase will throttle/timeout within 10 seconds
    FIX: Replace SSE polling with Redis pub/sub, add notification caching

9.4 Scenario: Apify Down for 24 Hours
    WHAT HAPPENS:
    - All tracking calls fail silently
    - Clips continue accumulating in tracking queue
    - No earnings updates for 24 hours
    - No views/likes/comments updates
    - Users see stale data but no errors
    - When Apify recovers: massive backlog, cron catches up over next few hours
    DATA LOSS: None. Just delayed updates.

9.5 Scenario: Supabase 1-Hour Outage
    WHAT HAPPENS:
    - ALL pages fail to load (db = null check, but no fallback UI)
    - All API routes return 500 errors
    - Cron tracking fails silently
    - SSE connections drop
    - Users see loading spinners or error states
    - When Supabase recovers: everything resumes normally
    DATA LOSS: None. But 1 hour of complete downtime for all users.
